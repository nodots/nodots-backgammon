WebSocket Connection Timeout and Fallback Test

Test Instructions

This test validates that the WebSocket reconnection and REST API fallback mechanisms work correctly when connections time out or drop.
  1. Start the game server with npm start
  2. Open the game in your browser and connect to a game
  3. Wait for the board to be inactive for more than 30 seconds to trigger a timeout
  4. Try clicking dice to pass turn or switch dice
  5. Verify that actions work via REST API fallback

Expected Behavior

✅ Before Fix: WebSocket timeout would cause dice clicks to throw errors and prevent turn passing
✅ After Fix: WebSocket timeout triggers automatic reconnection attempt, and if that fails, gracefully falls back to REST API

Key Improvements Made

  • Automatic Reconnection: When WebSocket disconnects, system attempts to reconnect before failing
  • Graceful Fallback: If reconnection fails, actions automatically use REST API instead of throwing errors
  • Fast E2E Recovery: Special handling for E2E test environments with immediate reconnection attempts
  • Better Error Handling: Clear logging and warning messages instead of critical errors

Test Validation

To validate the fix is working, check browser console logs for these messages:
  • WebSocket not connected - attempting reconnection before fallback
  • Reconnection failed - falling back to REST API
  • Still disconnected after reconnection attempt - using REST fallback
The key improvement is that instead of throwing an error saying "WebSocket connection required", the system now gracefully handles disconnections.

Manual Test Steps

  1. Start a game and get to a state where you can pass turn (rolled dice, moved pieces)
  2. Leave the browser tab inactive for 2-3 minutes
  3. Return to the tab and try to pass turn by clicking dice
  4. Action should succeed (either via reconnected WebSocket or REST fallback)
  5. Check console logs to see which path was taken
